## 1. 产品概述

### 1.1 背景

Claude Code 的 Skills 功能允许用户在文件系统中以 `.claude/` 目录 + `skill.md` 文件的形式定义可复用的 AI 能力片段。随着 Skills 数量增多，用户需要一个专用工具来：

- 统一发现和浏览分散在磁盘各处的 Skills
- 预览 Skill 内容及相关资源文件
- 在不同项目之间组织和搬移 Skills

### 1.2 用户定位

| 用户类型 | 描述 |
|---------|------|
| 主要用户 | 使用 Claude Code 的开发者，拥有 10–200 个 Skills |
| 次要用户 | 团队共享 Skills 库的技术负责人 |

### 1.3 约束条件

- 仅支持 macOS（Electron + Vite 构建）
- 不涉及网络传输，所有操作为本地文件系统操作
- 安全模型：`contextIsolation: true`，渲染层无直接 Node.js 访问

## 2. 功能需求

### FR-01 技能扫描与发现

| ID | 需求 | 优先级 |
|----|------|--------|
| FR-01-1 | 用户可指定一到多个根目录作为扫描范围（授权路径） | P0 |
| FR-01-2 | 系统使用 `find` 命令递归扫描授权路径下所有 `.claude/` 目录和 `skill.md` 文件 | P0 |
| FR-01-3 | 扫描深度可由用户配置（范围 2–15，默认 8） | P1 |
| FR-01-4 | 扫描过程中实时展示进度（当前路径、百分比、已发现数量） | P1 |
| FR-01-5 | 扫描完成后自动更新技能树，保持已展开节点状态 | P1 |
| FR-01-6 | 系统自动从 `skill.md` 或 `README.md` 提取技能名称和描述（首个 `#` 标题 + 首段落，≤200字符） | P1 |

### FR-02 技能分类

| ID | 需求 | 优先级 |
|----|------|--------|
| FR-02-1 | 系统按路径深度和特征将技能分为四个层级：计算机级、软件级、工作目录级、项目级 | P0 |
| FR-02-2 | 分类规则：`depth ≤ 2` → 计算机级；含 `/Application Support/` 或 `/Applications/` → 软件级；`depth ≤ 4` → 工作目录级；`depth > 4` → 项目级 | P0 |
| FR-02-3 | 每个层级在侧边栏作为独立分组展示，显示组内技能数量 | P1 |

### FR-03 技能树导航

| ID | 需求 | 优先级 |
|----|------|--------|
| FR-03-1 | 侧边栏以树形结构展示每个技能根目录及其子文件/子目录 | P0 |

## 3. 非功能需求

### NFR-01 性能

| ID | 需求 | 衡量标准 |
|----|------|---------|
| NFR-01-1 | 扫描 500 个目录的完成时间 ≤ 5 秒 | 实测 p95 |
| NFR-01-2 | 选中节点至预览内容可见的延迟 ≤ 200ms（文件 ≤ 1MB） | 实测 p95 |
| NFR-01-3 | 侧边栏滚动帧率 ≥ 60fps（节点数 ≤ 1000） | Chrome DevTools |
| NFR-01-4 | 应用冷启动（首帧可交互）≤ 3 秒 | 实测 p95 |
| NFR-01-5 | 内存占用（空闲状态）≤ 200MB | macOS Activity Monitor |

### NFR-02 安全性

| ID | 需求 |
|----|------|
| NFR-02-1 | 渲染进程不得拥有直接的 Node.js 访问权限（`nodeIntegration: false`） |
| NFR-02-2 | 所有渲染层调用 Node 能力必须经由 contextBridge 暴露的类型化 API |
| NFR-02-3 | 文件读取操作仅限于用户主动授权的路径范围内 |
| NFR-02-4 | 二进制文件通过 `skillsfile://` 协议加载，不得关闭 Web 安全策略 |
| NFR-02-5 | shell 命令中的用户输入路径须正确转义，防止命令注入 |

### NFR-03 可靠性

| ID | 需求 |
|----|------|
| NFR-03-1 | 权限不足的目录静默跳过，不影响整体扫描流程 |
| NFR-03-2 | 不存在的授权路径给出用户可感知的提示，而非静默忽略 |
| NFR-03-3 | 文件操作失败（复制/移动）须向用户展示明确错误信息 |
| NFR-03-4 | 扫描中断（进程崩溃）后重启应用可正常恢复至上次状态 |

### NFR-04 可用性

| ID | 需求 |
|----|------|
| NFR-04-1 | 符合 macOS HIG 规范：窗口管理、菜单栏、快捷键（`⌘R` 扫描、`⌘,` 设置） |
| NFR-04-2 | 深色/浅色模式视觉对比度满足 WCAG AA 标准（4.5:1） |
| NFR-04-3 | 全部交互操作可通过键盘完成（无障碍访问） |
| NFR-04-4 | 加载/进行中的操作须有明确的视觉反馈（加载指示器） |

### NFR-05 可维护性

| ID | 需求 |
|----|------|
| NFR-05-1 | 三进程（Main/Preload/Renderer）严格分离，Renderer 不包含 Node.js 依赖 |
| NFR-05-2 | 新增文件类型预览只需修改 3 处（scanner.ts、新建 Preview 组件、Preview/index.tsx） |
| NFR-05-3 | 新增 IPC 通道须同步更新 types/index.ts 的 IPC 常量 |
| NFR-05-4 | 新增 UI 文本须同步添加中英文翻译条目 |

### NFR-06 兼容性

| ID | 需求 |
|----|------|
| NFR-06-1 | 支持 macOS 12 Monterey 及以上版本 |
| NFR-06-2 | 同时提供 x64 和 arm64（Apple Silicon）构建产物 |
| NFR-06-3 | DMG 安装包无需 App Store，支持直接分发 |
